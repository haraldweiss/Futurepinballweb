## Executive Summary

This document provides comprehensive research on Future Pinball (Windows) FPT file texture specifications, including:
- Supported image formats and encoding methods
- Texture metadata and properties
- Material system and shader properties
- Texture extraction and handling in the web implementation
- Comparison with Visual Pinball (VPX) for context

**Key Finding**: Future Pinball stores textures as standard image formats (PNG, JPG, BMP, GIF, TGA, WebP) within OLE2/CFB container files, with automatic power-of-2 dimension scaling by the original engine.

## 1. Texture Formats Supported by Future Pinball

### Supported Image Formats

Future Pinball natively supports multiple image formats for texture import and storage:

| Format | Extension | Alpha Channel | Compression | Load Speed | Memory Usage |
|--------|-----------|---------------|-------------|------------|--------------|
| **BMP** | .bmp | Yes (32-bit) | Uncompressed | ~33% faster | ~33% faster load |
| **JPG/JPEG** | .jpg, .jpeg | No | Lossy (JPEG) | Standard | Same as BMP in VRAM |
| **TGA** | .tga | Yes | Uncompressed or RLE | Moderate | Same as BMP in VRAM |
| **GIF** | .gif | Yes | Lossless | Slower | Standard |
| **WebP** | .webp | Yes | Lossy/Lossless | Variable | Efficient |
| **PNG** | .png | Yes | Lossless | Moderate | Standard |

### Current Web Implementation Support

The Future Pinball web implementation (`src/fpt-parser.ts`) detects and extracts the following formats via magic byte identification:

```typescript
// Magic byte detection (lines 88-96)
PNG:  0x89 0x50 0x4E 0x47 (89 50 4E 47)
JPEG: 0xFF 0xD8
BMP:  0x42 0x4D (42 4D)
GIF:  0x47 0x49 0x46 (47 49 46)
WebP: 0x52 0x49 0x46 0x46 ... 0x57 0x45 (RIFF...WEBP)
```

### Format Recommendations

**For Quality**: TGA or PNG (with alpha channel support)
**For Performance**: BMP (fastest load time, ~33% quicker)
**For Compression**: JPG (smaller file size, lossy)
**For Transparency**: TGA or PNG (required if alpha needed)

**Critical Notes**:
- PNG is supported in the web version but not explicitly documented in original FP
- JPG uses lossy compression and **cannot store transparency**
- If transparency is needed, must use TGA or PNG
- BMP loaded significantly quicker than JPG in original FP testing

## 2. Texture Metadata and Properties

### Dimension Requirements (Power of 2)

The original Future Pinball engine has specific requirements for texture dimensions:

**Rule**: Texture dimensions should use **powers of 2**: 64, 128, 256, 512, 1024, 2048, 4096

**Examples of Valid Dimensions**:
- 512×512 ✓
- 1024×2048 ✓ (dimensions don't need to match)
- 256×512 ✓
- 128×1024 ✓

**Invalid Dimensions** (will be auto-scaled):
- 800×600 → Scaled to 1024×512 or 1024×1024
- 640×480 → Scaled to 512×512
- 512×600 → Scaled to 512×512 or 1024×1024

### Auto-Scaling Behavior

**In Original Engine**:
- Non-power-of-2 textures are automatically rescaled to nearest power-of-2 dimension
- **Performance Cost**: Rescaling happens at runtime, adds overhead to load times
- **Memory Impact**: Rescaled textures use more VRAM

**In Web Implementation**:
- Three.js handles power-of-2 conversion transparently
- No explicit scaling penalty (browser graphics APIs handle it)
- Textures are loaded via TextureLoader with sRGB colorspace applied

### Stored Metadata

**What IS Stored in FPT**:
- Raw image data (PNG/JPG/BMP/etc.)
- Texture name (derived from stream name in CFB structure)
- File format (detected via magic bytes)

**What IS NOT Stored** (derived at runtime):
- Actual pixel dimensions (calculated from image headers)
- Mipmap levels (generated by graphics engine)
- Compression format (implicit in image format)
- Usage context (playfield vs. backglass vs. translite)

**Stream Metadata in CFB**:
```
OLE2/CFB Entry
├─ name: "PlayfieldTexture" or similar
├─ size: Raw byte count
├─ content: Compressed or raw image data (LZO1X or raw)
└─ type: 2 (Stream, not storage)
```

### Texture Metadata Extraction Strategy

For optimal web extraction, the parser should capture:

1. **Texture Name**: From CFB stream name (e.g., "Playfield", "Backglass", "Translite")
2. **Format**: Detected via magic bytes
3. **Dimensions**: Extracted from image file headers
4. **Compression**: Implicit in image format
5. **Size**: Raw byte count from CFB entry

**Example Detection Heuristic** (from `fpt-parser.ts`):
```typescript
// Playfield texture = largest image
if (bytes.length > largestTexSize) {
  largestTexSize = bytes.length;
  fptResources.playfield = tex;
}

// Named texture extraction
if (/playfield|table|backdrop/i.test(nameL)) {
  fptResources.textures[name] = tex;
}
```

## 3. Material Properties and Shaders

### Material System in Original Future Pinball

The original FP uses **Newton Physics Engine** with **DirectX 9/OpenGL** rendering:

**Graphics Pipeline**:
- DirectX 9: Direct3D device with fixed-function or shader pipeline
- OpenGL: ARB assembly shaders or GLSL (depending on GPU support)
- Shader Compilation: Real-time compilation and caching

**DirectX Texture Compression**:
Original FP supports DXT compression (via DirectX):
- DXT1: RGB (no alpha) or 1-bit alpha
- DXT3: RGBA with explicit alpha (interpolated 4-bit)
- DXT5: RGBA with interpolated alpha (3-bit BC4)
- Format: Compressed on disk, decompressed for GPU use

### Material Properties Exposed to VBScript

**In Original Future Pinball VBScript**, materials support these properties:

```vbscript
' Material object properties (inferred from Newton engine)
Material.Color          ' RGB color value
Material.Shininess      ' 0.0-1.0 (specular power)
Material.Transparency   ' 0.0-1.0 (alpha)
Material.Ambient        ' Ambient RGB component
Material.Diffuse        ' Diffuse RGB component
Material.Specular       ' Specular RGB component
Material.EmissivePower  ' Emission intensity
```

### Current Web Implementation (Three.js PBR)

The web version uses **MeshStandardMaterial** (PBR = Physically Based Rendering):

```typescript
// From src/table.ts examples
const bumperMat = new THREE.MeshStandardMaterial({
  color: 0xff2200,              // Base color (albedo)
  roughness: 0.5,               // Surface roughness (0-1)
  metalness: 0.3,               // Metallic component (0-1)
  emissive: 0x330000,           // Emission color
  emissiveIntensity: 0.6,       // Emission strength
  transparent: true,            // Alpha support
  opacity: 0.8,                 // Alpha value
});
```

**PBR Properties Mapping**:

| Original FP | Three.js PBR | Description |
|-------------|--------------|-------------|
| Ambient | color/lightMap | Ambient contribution |
| Diffuse | color | Base color (albedo) |
| Specular | metalness, roughness | Reflectivity |
| Shininess | roughness | Surface smoothness (inverse) |
| Transparency | opacity, transparent | Alpha blending |
| Emissive | emissive, emissiveIntensity | Self-illumination |

### Normal Mapping and Advanced Properties

**Future Pinball Native Support**:
- **No explicit normal mapping** in original VBScript API
- Material properties are limited to basic Phong/Blinn model
- Normal mapping not directly exposed to table creators

**Web Implementation Capability**:
The web version can support normal mapping through:
- Extended material definitions (not in original FP)
- MS3D model extraction (includes normal data)
- Custom shader pipeline enhancements

**Potential Extensions** (for Phase 14+ Graphics Pipeline):
```typescript

## 4. Playfield Mesh Structure in FPT

### Playfield Geometry in FPT

**Stored Format**:
- Playfield surface is typically stored as:
  1. MS3D model file (Milkshape 3D binary format)
  2. Procedural description in binary metadata
  3. Ramp/lane definitions as curves

**MS3D Format** (documented in Phase 7):
```
Header: "MS3D" (4 bytes)
├─ Vertices (position, normal, UV, bone ID)
├─ Triangles (indices, per-vertex normals, UV)

### UV Mapping

**In Original FP**:
- UV coordinates typically range [0,1] × [0,1]
- Stored with vertex data in MS3D or procedural definitions
- Applied per-triangle to texture

### Playfield Coordinate System

**Future Pinball Units**:
- X-axis: -1050 to +1050 FP units (table width)
- Y-axis: 0 to 4200 FP units (table length, top to bottom)

## 5. Texture Extraction Tools and Methods

### In Current Web Implementation

**File**: `src/fpt-parser.ts` (1569 lines)

**Key Functions**:

1. **Magic Byte Detection** (lines 88-107):
   - `detectImageMime()`: Identifies image format
   - `detectAudioMime()`: Identifies audio format
   - Scans multiple offsets for hidden format markers

2. **Format Conversion** (lines 109-136):
   - `bytesToTexture()`: Converts bytes to Three.js Texture
   - `extractImageFromBytes()`: Tries multiple offsets and LZO decompression
   - Applies sRGB color space and clamp wrapping

3. **LZO Decompression** (lines 22-72):
   - `lzo1xDecompress()`: Full LZO1X-1 implementation
   - Handles compressed streams in FPT
   - ~200 line algorithm with proper error handling

4. **Resource Extraction** (lines 159-277):
   - `parseCFBResources()`: Main extraction function
   - Identifies textures by stream name pattern matching
   - Logs detailed extraction status

**Current Extraction Logic**:
```typescript
// Pattern matching for texture streams
if (/image|texture|playfield|table|backdrop|translite/i.test(nameL)) {
  const tex = await extractImageFromBytes(bytes);
  if (tex) fptResources.textures[name] = tex;
}

// Playfield = largest texture
if (bytes.length > largestTexSize) {
  fptResources.playfield = tex;
}
```

### Reverse Engineering Approach

**Techniques Used in Web Version**:

1. **CFB Stream Analysis**:
   - Scans OLE2/CFB directory for stream entries
   - Identifies resource types by name heuristics
   - Extracts by type (texture, sound, script)

2. **Magic Byte Scanning**:
   - Checks multiple offsets in stream data
   - Handles embedded headers (padding at start)
   - Supports both raw and LZO-compressed data

3. **Heuristic Classification**:
   - Stream name patterns: "Image*", "Texture*", "Playfield", "Backglass"
   - Size heuristics: Largest = playfield texture
   - Format detection: Magic bytes + fallback decoding

4. **Fallback Strategies**:
   - Try raw format detection first
   - Fall back to LZO decompression
   - Try multiple character encodings (UTF-8, UTF-16LE, ISO-8859-1)
   - Extract printable ASCII strings as last resort

## 6. Comparison: Future Pinball vs Visual Pinball Textures

### Texture Storage Approach

| Aspect | Future Pinball | Visual Pinball |
|--------|---|---|
| **Container** | OLE2/CFB | OLE2/CFB (same) |
| **Format Support** | PNG, JPG, BMP, GIF, TGA, WebP | PNG, JPG, BMP, TGA, DDS |
| **Compression** | DXT (DirectX) | DXT, JPEG, PNG |
| **Material Separation** | Material = Texture | Texture separate from material |
| **Metadata** | Minimal (name only) | Dimensions, format, usage hints |
| **PBR Support** | Limited (no normal maps) | Modern (supports PBR workflows) |
| **Export Format** | FPT (proprietary) | VPX (based on ZIP/OLE2) |

### Material Property Model

| Property | FP (Phong/Blinn) | VPX (PBR) |
|----------|---|---|
| **Base Color** | Diffuse RGB | Albedo/Color |
| **Reflectivity** | Specular RGB + Shininess | Metalness + Roughness |
| **Detail** | Normal (not exposed) | Normal map (explicit) |
| **Emission** | Emissive (basic) | Emissive map (advanced) |
| **Transparency** | Alpha (simple) | Alpha + thin transparency |
| **Environment** | Ambient (fixed) | Image-based lighting |

### Web Implementation Advantage

The web version (using Three.js PBR) is actually **more capable** than the original FP:

1. **Modern Material System**: PBR is more physically accurate
2. **Normal Mapping**: Not available in original FP
3. **Texture Atlasing**: Can optimize VRAM usage (Phase 14)
4. **Dynamic Lighting**: Per-pixel calculation vs. FP's simpler model
5. **Shader Quality**: Modern GLSL vs. FP's ARB assembly

## 7. Key Findings and Implications

### What Future Pinball Stores

✅ **Does Store**:
- Standard image files (PNG, JPG, BMP, GIF, TGA, WebP)
- Texture names (derived from CFB stream names)
- Basic material properties (color, shininess, transparency)
- MS3D 3D models with embedded materials
- Sound files and music tracks
- VBScript table code

❌ **Does NOT Store**:
- Normal maps (not supported in original VBScript)
- Roughness/Metalness maps
- Per-pixel material variation maps
- Explicit texture metadata (dimensions, etc.)
- Compressed DXT format on disk (only in memory via DirectX)

### For Web Implementation (Current Status)

**Already Implemented** ✅:
- Format detection (PNG, JPG, BMP, GIF, WebP)
- LZO decompression
- Three.js texture loading with sRGB
- Basic material properties
- MS3D model extraction
- Material system (9 types, 11 tuned interactions)

**Could Be Enhanced** 🔄:
- Normal map generation from textures (for Phase 14+)
- Texture atlasing for VRAM optimization
- Power-of-2 dimension warning/info
- Per-stream metadata extraction
- Advanced material property inference

**Web Advantages Over Original** 🚀:
- PBR materials (more realistic)
- Automatic mipmap generation
- GPU texture compression (ASTC, ETC2)
- HDR texture support
- WebGL texture wrapping options

## 8. Technical Specifications

### OLE2/CFB Stream Structure

```
FPT File (OLE2/CFB Container)
├─ Root Entry
│  ├─ Workbook
│  │  ├─ "Playfield" (stream) → Image data (PNG/JPG/BMP/GIF)
│  │  ├─ "Backglass" (stream) → Image data
│  │  ├─ "Translite" (stream) → Image data
│  │  ├─ "TableScript" (stream) → VBScript code (LZO compressed)
│  │  ├─ "Physics" (stream) → Physics parameters (XML or binary)
│  │  ├─ "Sound_*" (streams) → Audio files (WAV, OGG, MP3)
│  │  ├─ "MS3D_*" (streams) → 3D models (Milkshape format)
│  │  └─ Other metadata streams
│  └─ Other entries
└─ [Mini FAT for small streams]
```

### Image Data in Streams

**Raw Format** (no header):
```
Offset  Content
0x00    Magic bytes (0x89 0x50 0x4E 0x47 for PNG, etc.)
0x04    ... image data ...
0xNN    EOF
```

**With Padding** (offset varies):
```
Offset  Content
0x00    Padding (0x00 bytes or other)
0x08    Magic bytes
0x0C    ... image data ...
0xNN    EOF
```

**LZO Compressed**:
```
Offset  Content
0x00    Size hint (uint32_le, optional)
0x04    Compressed size (uint32_le, optional)
0x08    LZO header or data
0xNN    EOF (when decompressed)
```

### Texture Dimension Storage

**Not explicitly stored** in FPT stream metadata. Extracted from:

1. **PNG Header** (8 bytes after magic):
   ```
   Bytes 16-23: Width (uint32_be)
   Bytes 24-31: Height (uint32_be)
   ```

2. **JPEG Marker Scan**:
   - SOF (Start of Frame) marker contains dimensions

3. **BMP Header**:
   - Bytes 18-21: Width (uint32_le)
   - Bytes 22-25: Height (uint32_le)

## Conclusion

Future Pinball FPT files store textures as standard image formats within OLE2/CFB containers, detected via magic bytes and extracted heuristically. The original engine uses a Phong/Blinn lighting model with basic material properties, while the web version has upgraded to physically-based rendering with enhanced capabilities.

**Key Takeaway**: The web implementation can extract textures reliably from FPT files and actually provides *more advanced* material properties than the original through Three.js's PBR system, while maintaining compatibility with existing FPT table assets.
